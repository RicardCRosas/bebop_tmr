Muy buenas preguntas 👌 aquí ya estás tocando comportamiento real de ROS.

Vamos por partes.

---

# 🧠 1️⃣ ¿Por qué `rostopic echo /supervisor/state` no muestra nada si lo abro después?

Porque tu publisher es normal:

```python
self.state_pub = rospy.Publisher("/supervisor/state", String, queue_size=1)
```

Eso significa:

> Publica y olvida.

Si nadie estaba escuchando en ese momento → el mensaje se pierde.

---

# ✅ SOLUCIÓN: Publisher **latched**

En ROS1 puedes hacer que el tópico recuerde el último mensaje.

Solo cambia el publisher a:

```python
self.state_pub = rospy.Publisher(
    "/supervisor/state",
    String,
    queue_size=1,
    latch=True
)
```

🔥 Con `latch=True` pasa esto:

1. Publicas un estado.
2. Un nuevo suscriptor se conecta.
3. Automáticamente recibe el último estado publicado.

Ahora puedes:

```
rostopic echo /supervisor/state
```

Cerrar.
Volver a abrir.
Y siempre verás el estado actual.

Eso es exactamente lo que quieres.

---

# 🎯 2️⃣ ¿Por qué TAKEOFF ejecuta la acción antes de cambiar de estado?

Esto pasa porque tu transición se ve así:

```
state = TAKEOFF
↓
update()
↓
handle_takeoff()
↓
initial_takeoff()
↓
(espera)
↓
state = MISSION
```

Pero tu impresión del estado ocurre solo cuando `self.state` cambia.

El detalle importante:

Cuando llamas:

```python
self.supervisor.start_mission()
```

Ya hiciste:

```python
self.state = GlobalState.TAKEOFF
```

Pero el log solo se imprime cuando `update()` llama a `publish_state()`.

Y como tu acción ocurre dentro de `handle_takeoff()` antes de la siguiente iteración completa del timer, parece que la acción ocurrió primero.

En realidad el estado ya cambió internamente, solo que el log aparece después.

---

# 🔥 SOLUCIÓN CORRECTA Y PROFESIONAL

Cuando cambies estado, publícalo inmediatamente.

Haz esto:

---

## Paso 1 — Crear método de transición

Agrega esto en la clase:

```python
def change_state(self, new_state):

    if new_state == self.state:
        return

    self.state = new_state
    rospy.loginfo(f"STATE → {self.state.name}")
    self.state_pub.publish(self.state.name)
    self.last_state = self.state
```

---

## Paso 2 — Reemplaza TODOS los

```python
self.state = GlobalState.X
```

por

```python
self.change_state(GlobalState.X)
```

Ejemplo:

```python
self.change_state(GlobalState.TAKEOFF)
```

---

# 🎯 ¿Qué cambia ahora?

Ahora el flujo real es:

```
start_mission()
↓
change_state(TAKEOFF)
↓
SE IMPRIME INMEDIATAMENTE
↓
update()
↓
handle_takeoff()
↓
initial_takeoff()
```

Ahora sí:

Primero cambia estado.
Luego ejecuta acción.

Como debe ser en una FSM bien hecha.

---

# 🧠 Resumen final

✔ Usa `latch=True` para que el estado sea persistente
✔ Usa `change_state()` para transiciones limpias
✔ Nunca asignes `self.state = ...` directamente
✔ Publica estado en el momento de transición
